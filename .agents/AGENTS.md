# Backoffice Development Guidelines

เมื่อมีการพัฒนาหรือเพิ่มฟีเจอร์ใหม่ใน **ระบบหลังบ้าน (dh-backoffice-react)** ห้ามลืมทำสิ่งเหล่านี้เด็ดขาด:
1. **ต้องเพิ่มคำอธิบายและการสอนการใช้งานเข้าไปใน UI ของระบบด้วยทุกครั้ง** (In-App Documentation)
2. คู่มือ/คำอธิบายต้องประกอบไปด้วย:
   - **ตำรา / คำอธิบาย**: อธิบายว่าฟีเจอร์นี้คืออะไร ใช้ทำอะไร
   - **วิธีการใช้งาน (How-to)**: อธิบายเป็นขั้นเป็นตอน
   - **เทคนิคการใช้งาน (Tips & Tricks)**: บอกเทคนิคที่ทำให้ใช้งานได้ง่ายหรือเร็วขึ้น (เช่น Shortcut Keys)
   - **ตัวอย่างผลลัพธ์ (Expected Results)**: แสดงตัวอย่างหรือคำเตือนที่ชัดเจนก่อนผู้ใช้จะกดยืนยันการทำงาน

ให้คำนึงเสมอว่าผู้ใช้งานมีหลายระดับ (Staff ใหม่, Manager) การสร้าง UI ต้องมาพร้อมคำอธิบายในตัวเสมอ (เช่น Guide Modal, Tooltip, Help Panel)

# AI Core Directives (DH Notebook)

เพื่อให้ AI สามารถทำงานในโปรเจกต์ได้อย่างมีประสิทธิภาพสูงสุด กรุณายึดถือกฎต่อไปนี้ในทุกๆ การสนทนาและการพัฒนา:

1. **Auto-Refactor & SRP:** หากพบว่าไฟล์มีขนาดใหญ่, Logic ซับซ้อน หรือทำหลายหน้าที่ (Spaghetti Code) ให้ทำการ Extract Component, Custom Hook หรือ Service ออกมาใหม่ทันที โดยยึดหลัก Single Responsibility Principle
2. **Firebase Optimization:** การเขียนโค้ดต้องคำนึงถึงความสมดุลระหว่าง "ประสิทธิภาพ/ความเร็ว" และ "การประหยัดโควต้า Reads/Writes" อย่างเคร่งครัด
3. **Strict Data Relations:** ก่อนลบหรือปรับปรุงโครงสร้างข้อมูล ให้ตรวจสอบความสัมพันธ์ (Relations) กับเอกสารอื่นๆ เพื่อป้องกันปัญหาข้อมูลกำพร้าหรือระบบพัง
4. **Premium UX/UI:** ให้ความสำคัญกับความสวยงาม นำเสนอและประยุกต์ใช้ลูกเล่นที่ทำให้ระบบดูทันสมัยและใช้งานง่าย มุ่งสู่ความรู้สึก "High Quality System" เสมอ
5. **Safe Editing:** แก้ไขโค้ดเฉพาะส่วนที่ได้รับมอบหมาย ห้ามแก้ไขฟังก์ชันที่ไม่เกี่ยวข้อง และห้ามลบโค้ดเดิมที่ยังทำงานได้หรือไม่ได้สั่งให้ปรับปรุง
6. **Audit Log Awareness:** การเปลี่ยนแปลงข้อมูลสำคัญโดยเฉพาะที่เกิดจากผู้ใช้งาน (Staff/Manager) จะต้องมีระบบจัดการเพื่อบันทึกประวัติ (History Log) ด้วยเสมอ
