หลังจากทำการอุดรอยรั่ว Quota Leaks, เสริมความแข็งแกร่งด้าน Data Relationships (Atomic Sync) และ การอัปเกรดระบบ Promotions ล่าสุด (Best Promo + Real-time Validation)

## 1. 🧑‍💻 จำลองผู้ใช้งาน [ระบบหน้าบ้าน] (1,000 คน) [อัปเดตหลังทำ Frontend Audit]
เมื่อผู้ใช้งาน 1,000 คน (Active Users) เข้ามาใช้งานใน 1 วัน:
- **โหลดหน้าแรก & เข้าดูสินค้า (Smart Cache 24H):** เนื่องจากมีการขยายเวลา Cache อ่านข้อมูลส่วนใหญ่ผ่าน Memory Cache จำนวน Extra Reads จึงลดลงมหาศาล ~2 Reads / คน
- **ค้นหาและเลือกดูสินค้า:** (Limit Pagination) ดู 3-4 หน้า ~15 Reads / คน
- **พฤติกรรมการหยิบลงตะกร้า & เข้าหน้า Cart (30% ของผู้ใช้):** 
  - การโหลดหน้า Cart มีการทำ Real-time SKU Validation = ดึงสต๊อกล่าสุด (สมมติเฉลี่ย 3 SKU) = 3 Reads
  - ดึงข้อมูล Active Promotions (ด้วย Cache 5 นาที) = 1 Read (หารเฉลี่ย)
  - อัปเดตตะกร้า = 2 Writes (Batch Merge)
  - รวม: ~4 Reads / ~2 Writes / คน (เฉพาะ 300 คน)
- **พฤติกรรมการสั่งซื้อสำเร็จ (10% ของผู้ใช้):** (สร้าง Order, ตัดสต๊อก, เก็บประวัติ, **ดึงข้อมูลโปรโมชันแบบ Real-time (Transaction Validation) +1 Read**, และสร้าง Todo) ~7 Reads / ~9 Writes / คน (เฉพาะ 100 คน)
- **พฤติกรรมการใช้งาน สินค้าที่ถูกใจ (Favorites & Notes) (20% ของผู้ใช้):** (อ่านข้อมูลใช้ Cache ของ User Profile = 0 Extra Read, อัปเดต Note/Tags) ~0 Reads / ~2 Writes / คน (เฉพาะ 200 คน)

- **การโหลดหน้า Store Profile (โฆษณา & รีวิว) (100% ของผู้ใช้):** 
  - การโหลดหน้าใช้ `getUserPartnerAds` (ดึง 3 collections: partner_ads, user_sku_ads, billboard_ads) = 3 Reads
  - การโหลด Reviews (onSnapshot) เฉลี่ย 5 คอมเมนต์ = 5 Reads
  - รวม: ~8 Reads / คน
- **การพิมพ์รีวิว & กดหัวใจ (10% ของผู้ใช้):** (สร้างรีวิว หรือ อัปเดต like) = 1 Write / คน (เฉพาะ 100 คน)

**🔥 คำนวณผู้ใช้งาน 1,000 คน / วัน (ใหม่):**
- **รวม Reads:** (1000 x 15) + (300 x 4) + (100 x 7) + (1000 x 8) = 15,000 + 1,200 + 700 + 8,000 = **24,900 Reads**
- **รวม Writes:** (300 x 2) + (100 x 9) + (200 x 2) + (100 x 1) = 600 + 900 + 400 + 100 = **2,000 Writes**

## 2. 👨‍💼 จำลองพนักงาน [ระบบหลังบ้าน] (10 คน ทำงาน 09:00 - 18:00) [อัปเดตหลังเปิดระบบ Local Caching 100%]
สมมติพนักงาน 10 คน ทำงานเต็มวันบนหน้าเว็บ Backoffice โดยใช้คอมพิวเตอร์บริษัท:

  - ข้อมูลจะถูกดึงจาก Server (Reads) เฉพาะตอนมีข้อมูลใหม่ (Delta) หรือ Background Sync เมื่อหมดเวลา Stale Time (5 นาที)

- **รวมพนักงาน 1 คน:** ใช้ประมาณ **400 Reads | 250 Writes**

**🔥 คำนวณพนักงาน 10 คน / วัน (ใหม่สุด):**

- **รวม Writes:** (10 x 250) + 100 = **2,600 Writes**

## 4. 📉 เปรียบเทียบกับระบบก่อนหน้า (Quota Leak)
- **ก่อนการแก้ Quota Leaks:** หน้า Todo ดึงงาน 100 ล่าสุด (ซึ่งอาจเป็นงาน Completed ที่ถูกดันตกขอบ) แล้วต้องโหลดใหม่เรื่อยๆ แถมดึงงานคนละ Role (Manager ดึงของ Staff) ทำให้ Reads เสียเปล่ากว่า 50% ต่อการโหลด 1 ครั้ง
- **หลังอัปเกรด Audit:** ปรับระบบให้ Query เฉพาะ Status ที่ `pending` จาก Server เท่านั้น และปรับ Limit ให้รองรับปริมาณงานที่ตกขอบได้สูงขึ้นอย่างสมเหตุสมผล 
- **การเพิ่ม Security Validation (Promotions):** ถึงแม้จะมีการดึงข้อมูลโปรโมชันแบบ Real-time ณ จังหวะชำระเงิน แต่เนื่องจากเราย้ายการคำนวณ Best Promo และดึงแค่ 1 Document ทำให้ยอด Reads เฉลี่ยต่อออเดอร์ถูกควบคุมได้ดีกว่าเดิม การจำกัด Quota จึงมีความรัดกุม 100% โดยไม่ทำให้โควต้าบานปลาย

## 6. 📝 ประเมินโควต้าเพิ่มเติม: ระบบตีกลับสลิป (Reject Slip Reason) [ใหม่]
เมื่อมีการเพิ่มการบังคับใส่เหตุผลในการปฏิเสธสลิปโอนเงิน (Reject Slip):
- **จำนวน Reads:** ไม่มีการอ่านข้อมูลเพิ่ม (ใช้การอ้างอิง Document Reference เดิมที่มีอยู่แล้วในหน่วยความจำ) = **0 Extra Reads**
- **จำนวน Writes:** การอัปเดตสถานะออเดอร์ (`orders`), อัปเดตงาน (`todos`), และสร้างประวัติแจ้งลูกค้า (`gasHistory`) เกิดขึ้นแบบ Batch = **3 Writes / การปฏิเสธ 1 ครั้ง**
- **ผลกระทบรวมต่อวัน:** หากมีออเดอร์ถูกปฏิเสธสลิป 100 ออเดอร์/วัน (สมมติฐาน 10% ของ 1,000 users) 
  - = 100 x 3 = **300 Writes** (เพิ่มขึ้นเพียงเล็กน้อยมาก เมื่อเทียบกับโควต้า 20,000 Writes/วัน)
  - **ลดความผิดพลาด (Accuracy):** สร้างประวัติการทำรายการที่มีความแม่นยำสูง (Audit Trail)
  - **สร้างความโปร่งใส:** ทุกการแก้ไขมีประวัติชัดเจน ลดปัญหาข้อขัดแย้งระหว่างพนักงานกับลูกค้า

## 9. 🎨 ประเมินโควต้าเพิ่มเติม: ระบบหน้าบ้าน UX/UI Upgrade (Real-time & Cross-selling) [ใหม่ล่าสุด]
อัปเดตหลังจากเพิ่มระบบ Real-time ให้หน้ารายละเอียดสินค้า, สินค้าที่เกี่ยวข้อง (Related Products) และการดึงข้อมูลตะกร้าแบบ Batch (in query)

**จำลองผู้ใช้งาน 1,000 คนบนหน้าบ้าน:**
- **Product Detail (Real-time `onSnapshot`):** สมมติคนเข้า 5 ครั้ง/วัน 
  - 1 โหลดแรก + 1 การเปลี่ยนราคา/สต๊อกระหว่างดู = 2 Reads/ครั้ง
  - **Related Products:** คิวรีหมวดหมู่เดียวกัน (`limit(5)`) = 5 Reads/ครั้ง
  - รวม: 7 Reads x 5 ครั้ง = 35 Reads / คน
- **Cart (Batch Fetching):** สมมติมี 3 สินค้าในตะกร้า
  - โหลดหน้าตะกร้า ใช้ `getProductsByIds` (where in) = 3 Reads
  - กดสั่งซื้อ (Checkout) โหลดเช็คสต๊อกอีกครั้ง = 3 Reads
  - รวม: 6 Reads / คน (เดิมอ่านแบบ Promise.all ก็ 6 Reads เช่นกัน แต่ประหยัดเวลาและ Connection มากขึ้น)
- **Home & Search:** คงเดิมเฉลี่ย ~75 Reads / คน
- **รวม Reads หน้าบ้านต่อคน (หลัง Upgrade):** 35 + 6 + 75 = **116 Reads / คน / วัน**

**🔥 คำนวณผู้ใช้งาน 1,000 คน / วัน (ใหม่หลัง Upgrade):**
- **รวม Reads ทั้งหมด:** (1,000 คน x 116 Reads) + 4,000 (หลังบ้าน) = **120,000 Reads / วัน**
- **รวม Writes ทั้งหมด:** **4,600 Writes / วัน** (คงเดิม ไม่เปลี่ยนแปลงจากการอัปเกรดหน้าบ้าน)

**💰 ประเมินค่าใช้จ่าย (Cost Estimation หลัง Upgrade):**
- **Writes:** ไม่เกินโควต้าฟรี 20,000 ➔ **0 บาท**
- **Reads:** 120,000 - 50,000 (ฟรี) = **70,000 Reads ส่วนเกิน**
- **ราคา Firestore Blaze:** $0.03 (1 บาท) ต่อ 100,000 Reads ส่วนเกิน
- **สรุปรายเดือน:** **~21 บาท / เดือน!!** (1 บาท x 30 วัน x 0.7) สำหรับผู้ใช้งาน 1,000 คนต่อวัน ได้ระบบเทียบเท่า Lazada/Shopee (Real-time, Auto-scroll, Related Products) ซึ่งถือว่าคุ้มค่าการลงทุนสุดๆ!

## 7. 📦 ประเมินโควต้าเพิ่มเติม: ระบบอัปเดตสถานะจัดส่ง/มอบสินค้า (Shipping Management) [ใหม่]
เมื่อพนักงานทำการอัปเดตสถานะจาก "เตรียมจัดส่ง" เป็น "จัดส่งแล้ว" หรือ "ส่งมอบสำเร็จ":
- **จำนวน Reads:** ดึงข้อมูลบิล 1 Document สำหรับ Transaction = **1 Extra Read / การจัดส่ง**
- **จำนวน Writes:** 
  - การอัปเดตข้อมูลการจัดส่งใน `orders` (ใส่ Tracking No. หรือเปลี่ยนสถานะ) = 1 Write
  - การเก็บบันทึกประวัติ `historyService` = 1 Write

- **ผลกระทบรวมต่อวัน:** หากมีบิลที่ถูกส่งมอบ 100 ออเดอร์/วัน (สมมติฐาน 10% ของ 1,000 users) 
  - = 100 x 1 = **100 Reads**
  - = 100 x 2 = **200 Writes**
- **สรุป:** ส่งผลกระทบต่อโควต้าเพียงเล็กน้อยมาก (เพิ่ม Reads แค่ 100 ครั้ง และ Writes แค่ 200 ครั้ง) ภายใต้ฟังก์ชันที่เสริมประสิทธิภาพให้ฝ่ายจัดส่งทำงานได้สมบูรณ์ขึ้น

## 8. 🔍 ประเมินโควต้าเพิ่มเติม: ระบบดักจับความเปลี่ยนแปลงขั้นสูง (Advanced Change Detection Snapshot) [ใหม่]
เมื่อพนักงานใช้งานแผงสรุปความเปลี่ยนแปลง (Change Summary Panel) เพื่อเปรียบเทียบก่อนส่งออกสต็อกให้ Big Seller:
- **จำนวน Reads:** 
  - การโหลดหน้าจอ Generate Sync จะดึง Snapshot ล่าสุด (`system_counters/last_detect_snapshot`) = **1 Read / ครั้ง**
  - ด้วยระบบ Auto Sync ทุกๆ 15 นาที พนักงาน 10 คน ทำงาน 8 ชั่วโมง (480 นาที) = (480/15) * 10 = **~320 Reads / วัน**
  - ส่วนการเปรียบเทียบความแตกต่างระหว่างสต็อก Firebase (ข้อมูลดิบ) และ GAS (Sheet) **ใช้ 0 Reads** (ทำงานผ่าน Local Memory)
- **จำนวน Writes:** 
  - เมื่อพนักงานกด "บันทึกการดักจับ" ระบบจะสร้าง Transaction ภายในและอัปเดต Counter (`system_counters`) = **1 Write / การกดบันทึก**
  - ข้อมูลความเปลี่ยนแปลงเชิงลึกทั้งหมด (Detailed Array) จะถูกยิงตรงเข้า Google Sheets ผ่าน `gasHistoryService` = **0 Writes**
  - สมมติพนักงานทำการบันทึกเพื่อตรวจสอบความเรียบร้อยเฉลี่ย 5 ครั้ง/วัน = 5 x 1 = **5 Writes / วัน**
- **สรุป:** การเพิ่มระบบตัวดักจับขั้นสูงที่แสดง Transaction ID ภายในและ Badge เปรียบเทียบความถูกต้องนี้ ส่งผลให้เกิด **Reads เพิ่มขึ้นเพียง ~320/วัน และ Writes เพียง ~5/วัน** ซึ่งถือว่าประหยัดโควต้าในระดับสูงสุด (Highly Optimized) โดยยังคงให้ประสบการณ์การทำงานของพนักงานที่ยอดเยี่ยมและมั่นใจได้ 100% ว่าข้อมูลส่งออกถูกต้อง พร้อมส่ง History Log แบบละเอียดให้ Analyst ใน Google Sheets แบบฟรีๆ

## 9. 🚀 ประเมินโควต้าเพิ่มเติม: ระบบค้นหาสินค้าหน้า POS (Hybrid Cache Optimization) [อัปเดตล่าสุด]
เมื่อปรับใช้สถาปัตยกรรม Hybrid Cache (ดึงข้อมูล 5,000+ รายการจาก Google Sheets มาไว้ใน SessionStorage) ในหน้า POS:
- **ก่อนทำ (Legacy POS Search):** ทุกครั้งที่พนักงานพิมพ์ค้นหาสินค้า จะเกิด Firebase Query และเสีย Quota เฉลี่ย 2-5 Reads ต่อการค้นหา 1 ครั้ง ถ้า 1 วันค้นหา 1,000 ครั้ง จะเสีย **~5,000 Reads**
- **หลังทำ (Zero-Reads Architecture):**
  - **จำนวน Reads สำหรับค้นหาสินค้า:** ดึงเฉพาะ 50 ชิ้นแรกเป็นค่าตั้งต้น 1 ครั้งตอนเปิดหน้าต่าง POS = **1 Read Query** 
  - การพิมพ์ค้นหาสินค้าหลังจากนั้นทำงานบน Local Memory 100% = **0 Reads** ตลอดทั้งวัน
- **สรุป:** การอัปเกรดนี้ทำให้ POS เร็วปานสายฟ้า (0ms Latency) ลดอาการกระตุกตอนพนักงานขายหน้าร้าน และ**ประหยัดโควต้าได้ถึง 100% สำหรับการค้นหา** ทำให้ยอดรวมการใช้ Reads ของพนักงาน 10 คน ยิ่งลดลงกว่าเดิม มั่นใจได้ว่าอยู่ใต้ Free Tier อย่างสมบูรณ์แบบ

## 10. 🛡️ ประเมินโควต้าเพิ่มเติม: ระบบตรวจสอบคำขอผู้จัดการอัตโนมัติ (Manager Auto-Verification) [ใหม่ล่าสุด]
จากการพัฒนาระบบตรวจสอบและทำความสะอาดคำขอที่ค้างในแถบ **Manager Approvals** (เช่น คำขอโฆษณาที่ถูกอนุมัติหรือลบไปแล้วจากเมนูอื่น):
- **จำนวน Reads:** ระบบทำงานด้วยการฝังตัวอยู่ใน Component UI (Formal Cards) เมื่อการ์ดแต่ละใบถูกโหลดขึ้นมา จะทำการยิงดึงข้อมูล (1 Read) เฉพาะเอกสารเป้าหมาย (เช่น `partner_ads`) ของคิวนั้นๆ
  - ใน 1 วัน ผู้จัดการอาจเปิดหน้านี้เฉลี่ย 10 ครั้ง สมมติมีคำขอค้างอยู่ 5 รายการ = 10 x 5 = **~50 Reads / วัน**
- **จำนวน Writes:** เมื่อตรวจพบว่าเอกสารเป้าหมายหายไป (Not Exists) หรือถูกประมวลผลไปแล้ว (Active/Rejected) ระบบจะทำการอัปเดตงานในคิว (Todos) ให้กลายเป็น `completed` แบบเงียบๆ (Silent Resolve) = **1 Write / คิวที่ถูกเคลียร์**
- **สรุป:** การใช้ **"UI-Level Auto-Verification"** แทนการยิง Background Cron Jobs หรือ Cloud Functions ทำให้ **Reads เสียไปเพียงเล็กน้อย (หลักสิบ) เมื่อผู้จัดการเข้าใช้งานจริงเท่านั้น** ถือว่าเป็นการแก้ไขบั๊คข้อมูลค้าง/ขยะสะสม ที่ชาญฉลาดและประหยัดโควต้า 100% ตามแนวทาง SRP (Single Responsibility Principle)

## 13. 🔍 ประเมินโควต้าเพิ่มเติม: System-Wide Reads/Writes Quota Leak Audit [ใหม่ล่าสุด]
จากการทำ Audit ค้นหารอยรั่วของการสแกนและจัดการข้อมูลทั้งระบบ (System-Wide Audit):
- **1. ปัญหาการทำ SKU Validation ใน `freebieService.js`:**
  - **ก่อนแก้ไข:** หากมีแอดมินใส่ SKU สินค้า 1,000 ถึง 10,000 รายการ ระบบจะทำ `in` queries รอบละ 30 (333 รอบ) ทำให้กินโควต้า Reads ไปถึง 10,000 Reads ทันทีที่อัปเดตของแถม 
  - **หลังแก้ไข:** สร้างระบบ Bypass หาก SKU เกิน 500 รายการ จะตัดการตรวจเช็คฐานข้อมูลและเตือนให้ผู้ดูแลระบบทราบ เพื่อป้องกัน **Massive Read Leaks** (ประหยัด 10,000 Reads ลงเหลือ 0)
- **2. ปัญหาการทำ Chunk 10 ชิ้นในระบบอนุมัติราคาส่ง (Wholesale):**
  - **ก่อนแก้ไข:** ตัดแบ่งรอบค้นหาราคา 10 ชิ้นต่อ Firebase Query ทำให้เสียจำนวน Queries มากเกินควร
  - **หลังแก้ไข:** ปรับเป็น 30 ชิ้น (เต็ม Max Limit ของ `in` array) ลดจำนวน Queries ลงได้ถึง 3 เท่า
- **3. ปัญหา Orphaned Todos Cleanup (Massive Reads):**
  - **ก่อนแก้ไข:** ดึงข้อมูล งาน (Todos) สถานะ Pending ทั้งหมดในระบบ (อาจสูงถึงหลายพัน) มากรองแบบ In-memory เวลาที่ระงับ/ลบบัญชีลูกค้า
  - **หลังแก้ไข:** คิวรีเจาะจงเฉพาะ `customerUid` หรือ `payload.customerUid` ผ่าน `Promise.all` ทำให้จำนวน Reads ลดจาก "ดึงทั้งหมด" (O(N)) เหลือเพียงเฉพาะ 1-3 งานของลูกค้ารายนั้นๆ เท่านั้น (O(1)) ประหยัดหลักพัน Reads ต่อการแก้ไขโปรไฟล์ลูกค้า 1 ครั้ง
- **สรุป (Impact):** การปิดรอยรั่วขั้นลึกในระดับ System-level ครั้งนี้ ถือเป็นการการันตีว่าระบบจะไม่มีเพดานค่าใช้จ่ายที่บานปลายแบบเฉียบพลัน (Spike) แม้จะมีฐานข้อมูลขยายตัวไปที่หลัก 10,000 - 100,000 เอกสารก็ตาม
