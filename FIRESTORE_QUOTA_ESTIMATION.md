# =====================================================================
# FIRESTORE QUOTA SIMULATION & ESTIMATION (After Quota Leak Audit)
# =====================================================================

เอกสารนี้วิเคราะห์และจำลองการใช้งาน (Simulation) การใช้โควต้า (Reads/Writes) ของระบบใน 1 วัน (24 ชั่วโมง) 
หลังจากทำการอุดรอยรั่ว Quota Leaks, เสริมความแข็งแกร่งด้าน Data Relationships (Atomic Sync) และ การอัปเกรดระบบ Promotions ล่าสุด (Best Promo + Real-time Validation)

---

## 1. 🧑‍💻 จำลองผู้ใช้งาน [ระบบหน้าบ้าน] (1,000 คน) [อัปเดตหลังทำ Frontend Audit]
เมื่อผู้ใช้งาน 1,000 คน (Active Users) เข้ามาใช้งานใน 1 วัน:
- **โหลดหน้าแรก & เข้าดูสินค้า (Smart Cache 24H):** เนื่องจากมีการขยายเวลา Cache อ่านข้อมูลส่วนใหญ่ผ่าน Memory Cache จำนวน Extra Reads จึงลดลงมหาศาล ~2 Reads / คน
- **ค้นหาและเลือกดูสินค้า:** (Limit Pagination) ดู 3-4 หน้า ~15 Reads / คน
- **พฤติกรรมการหยิบลงตะกร้า & เข้าหน้า Cart (30% ของผู้ใช้):** 
  - การโหลดหน้า Cart มีการทำ Real-time SKU Validation = ดึงสต๊อกล่าสุด (สมมติเฉลี่ย 3 SKU) = 3 Reads
  - ดึงข้อมูล Active Promotions (ด้วย Cache 5 นาที) = 1 Read (หารเฉลี่ย)
  - อัปเดตตะกร้า = 2 Writes (Batch Merge)
  - รวม: ~4 Reads / ~2 Writes / คน (เฉพาะ 300 คน)
- **พฤติกรรมการสั่งซื้อสำเร็จ (10% ของผู้ใช้):** (สร้าง Order, ตัดสต๊อก, เก็บประวัติ, **ดึงข้อมูลโปรโมชันแบบ Real-time (Transaction Validation) +1 Read**, และสร้าง Todo) ~7 Reads / ~9 Writes / คน (เฉพาะ 100 คน)
- **พฤติกรรมการใช้งาน สินค้าที่ถูกใจ (Favorites & Notes) (20% ของผู้ใช้):** (อ่านข้อมูลใช้ Cache ของ User Profile = 0 Extra Read, อัปเดต Note/Tags) ~0 Reads / ~2 Writes / คน (เฉพาะ 200 คน)

- **การโหลดหน้า Store Profile (โฆษณา & รีวิว) (100% ของผู้ใช้):** 
  - การโหลดหน้าใช้ `getUserPartnerAds` (ดึง 3 collections: partner_ads, user_sku_ads, billboard_ads) = 3 Reads
  - การโหลด Reviews (onSnapshot) เฉลี่ย 5 คอมเมนต์ = 5 Reads
  - รวม: ~8 Reads / คน
- **การพิมพ์รีวิว & กดหัวใจ (10% ของผู้ใช้):** (สร้างรีวิว หรือ อัปเดต like) = 1 Write / คน (เฉพาะ 100 คน)

**🔥 คำนวณผู้ใช้งาน 1,000 คน / วัน (ใหม่):**
- **รวม Reads:** (1000 x 15) + (300 x 4) + (100 x 7) + (1000 x 8) = 15,000 + 1,200 + 700 + 8,000 = **24,900 Reads**
- **รวม Writes:** (300 x 2) + (100 x 9) + (200 x 2) + (100 x 1) = 600 + 900 + 400 + 100 = **2,000 Writes**

## 2. 👨‍💼 จำลองพนักงาน [ระบบหลังบ้าน] (10 คน ทำงาน 09:00 - 18:00)
สมมติพนักงาน 10 คน ทำงานเต็มวันบนหน้าเว็บ Backoffice:
- **หน้าศูนย์ปฏิบัติการ (Central Todo):** 
  - โหลดหน้าแรกดึงงาน Pending (Limit 300) = 300 Reads/โหลด
  - พนักงาน 1 คนเปิดหน้า Todo หรือรีเฟรชวันละ 10 ครั้ง = 3,000 Reads/คน
- **เมนู Search & Billing:** 
  - ค้นหาออเดอร์/บิล (Debounced Search) 20 ครั้ง/วัน = 400 Reads/คน
- **การอนุมัติสลิป/จัดการสต็อก (Writes):**
  - อนุมัติ/ปฏิเสธงานประมาณ 50 งาน/คน (มี Transaction อัปเดตสถานะ, สถิติยอดขาย) = 150 Writes/คน
  - **หัก Quota โปรโมชัน (เฉพาะบิลที่ Approved):** บิลละ 1 Write x 50 งาน = 50 Writes/คน
  - การแก้ไขราคาส่ง/ตรวจสอบ = 50 Writes/คน
- **รวมพนักงาน 1 คน:** ใช้ประมาณ **3,400 Reads | 250 Writes**

**🔥 คำนวณพนักงาน 10 คน / วัน:**
- **รวม Reads:** 10 x 3,400 = **34,000 Reads**
- **รวม Writes:** (10 x 250) + 100 = **2,600 Writes**

## 3. 📊 สรุปการใช้โควต้าทั้งหมดใน 24 ชั่วโมง (หน้าบ้าน + หลังบ้าน) [อัปเดต]
- **TOTAL READS:** 24,900 + 34,000 = **58,900 Reads / วัน**
- **TOTAL WRITES:** 2,000 + 2,600 = **4,600 Writes / วัน** (รองรับการ Track Promotion Quota, Ad Click, History Logs, และ Reviews)

## 4. 📉 เปรียบเทียบกับระบบก่อนหน้า (Quota Leak)
- **ก่อนการแก้ Quota Leaks:** หน้า Todo ดึงงาน 100 ล่าสุด (ซึ่งอาจเป็นงาน Completed ที่ถูกดันตกขอบ) แล้วต้องโหลดใหม่เรื่อยๆ แถมดึงงานคนละ Role (Manager ดึงของ Staff) ทำให้ Reads เสียเปล่ากว่า 50% ต่อการโหลด 1 ครั้ง
- **หลังอัปเกรด Audit:** ปรับระบบให้ Query เฉพาะ Status ที่ `pending` จาก Server เท่านั้น และปรับ Limit ให้รองรับปริมาณงานที่ตกขอบได้สูงขึ้นอย่างสมเหตุสมผล 
- **การเพิ่ม Security Validation (Promotions):** ถึงแม้จะมีการดึงข้อมูลโปรโมชันแบบ Real-time ณ จังหวะชำระเงิน แต่เนื่องจากเราย้ายการคำนวณ Best Promo และดึงแค่ 1 Document ทำให้ยอด Reads เฉลี่ยต่อออเดอร์ถูกควบคุมได้ดีกว่าเดิม การจำกัด Quota จึงมีความรัดกุม 100% โดยไม่ทำให้โควต้าบานปลาย

## 5. 💰 ประเมินค่าใช้จ่ายหากเกินโควต้า (Cost Estimation)
Firebase (Spark Plan) ให้โควต้าฟรีต่อวันอยู่ที่: 
- Reads: 50,000 / วัน 
- Writes: 20,000 / วัน

**ส่วนเกิน (Over Quota):**
- **Writes:** 4,600 (ไม่เกินโควต้าฟรี 20,000) ➔ **0 บาท**
- **Reads:** 58,900 - 50,000 = **8,900 Reads ส่วนเกิน**
- **ราคา Firestore Blaze:** $0.03 (ประมาณ 1 บาท) ต่อ 100,000 Reads ส่วนเกิน

**สรุปค่าใช้จ่ายจริง:**
- 8,900 Reads ส่วนเกิน = **$0.0026 ต่อวัน** (ประมาณ 0.09 บาท / วัน)
- **สรุปรายเดือน:** เดือนละ **2.7 บาท** ต่อการมีผู้ใช้งาน 1,000 คนต่อวัน!

** บทสรุป ** 
ด้วยการออกแบบและการแก้ไขล่าสุด รวมกับการเพิ่มฟีเจอร์ Reviews & Ads ระบบสามารถสเกลรับผู้ใช้งานระดับ 1,000 - 5,000 Active Users ต่อวันได้โดยมีต้นทุน Firestore เพียงไม่กี่บาทต่อเดือนครับ

## 6. 📝 ประเมินโควต้าเพิ่มเติม: ระบบตีกลับสลิป (Reject Slip Reason) [ใหม่]
เมื่อมีการเพิ่มการบังคับใส่เหตุผลในการปฏิเสธสลิปโอนเงิน (Reject Slip):
- **จำนวน Reads:** ไม่มีการอ่านข้อมูลเพิ่ม (ใช้การอ้างอิง Document Reference เดิมที่มีอยู่แล้วในหน่วยความจำ) = **0 Extra Reads**
- **จำนวน Writes:** การอัปเดตสถานะออเดอร์ (`orders`), อัปเดตงาน (`todos`), และสร้างประวัติแจ้งลูกค้า (`gasHistory`) เกิดขึ้นแบบ Batch = **3 Writes / การปฏิเสธ 1 ครั้ง**
- **ผลกระทบรวมต่อวัน:** หากมีออเดอร์ถูกปฏิเสธสลิป 100 ออเดอร์/วัน (สมมติฐาน 10% ของ 1,000 users) 
  - = 100 x 3 = **300 Writes** (เพิ่มขึ้นเพียงเล็กน้อยมาก เมื่อเทียบกับโควต้า 20,000 Writes/วัน)
- **สรุป:** การปรับปรุงนี้ช่วยเพิ่มความสมบูรณ์ในการสื่อสารกับลูกค้า (Customer Experience) โดยไม่ส่งผลกระทบที่มีนัยสำคัญต่อโควต้าและค่าใช้จ่ายของระบบแต่อย่างใด

## 7. 📦 ประเมินโควต้าเพิ่มเติม: ระบบอัปเดตสถานะจัดส่ง/มอบสินค้า (Shipping Management) [ใหม่]
เมื่อพนักงานทำการอัปเดตสถานะจาก "เตรียมจัดส่ง" เป็น "จัดส่งแล้ว" หรือ "ส่งมอบสำเร็จ":
- **จำนวน Reads:** ดึงข้อมูลบิล 1 Document สำหรับ Transaction = **1 Extra Read / การจัดส่ง**
- **จำนวน Writes:** 
  - การอัปเดตข้อมูลการจัดส่งใน `orders` (ใส่ Tracking No. หรือเปลี่ยนสถานะ) = 1 Write
  - การเก็บบันทึกประวัติ `historyService` = 1 Write
  - **รวม = 2 Writes / การจัดส่ง**
- **ผลกระทบรวมต่อวัน:** หากมีบิลที่ถูกส่งมอบ 100 ออเดอร์/วัน (สมมติฐาน 10% ของ 1,000 users) 
  - = 100 x 1 = **100 Reads**
  - = 100 x 2 = **200 Writes**
- **สรุป:** ส่งผลกระทบต่อโควต้าเพียงเล็กน้อยมาก (เพิ่ม Reads แค่ 100 ครั้ง และ Writes แค่ 200 ครั้ง) ภายใต้ฟังก์ชันที่เสริมประสิทธิภาพให้ฝ่ายจัดส่งทำงานได้สมบูรณ์ขึ้น
